**Template list:** `useQuery({ queryKey: ['templates'], queryFn: () => fetch('/api/templates').then(r => r.json()) })`. Display as a list/table (Claude's discretion per CONTEXT.md) showing: template title, default urgency (if set), default effort estimate (if set), created date. **Create template:** A "New Template" button opens a shadcn Sheet or Dialog containing a form with fields:
- title: text input (required)
- problem_statement: Tiptap editor (optional, `immediatelyRender: false`)
- default_urgency: number 1–5 (optional)
- default_effort_estimate: number (optional)
- default_next_step: textarea (optional)

**Edit template:** Clicking "Edit" opens the same form pre-filled with existing values. On submit: PATCH /api/templates/{id} with changed fields, invalidate ['templates'], close dialog.

**Delete template:** Clicking "Delete" opens a shadcn AlertDialog "Delete template '{title}'? This cannot be undone." On confirm: DELETE /api/templates/{id}, invalidate ['templates'].

Use react-hook-form + zod for the create/edit form. Schema: `z.object({ title: z.string().min(1), problem_statement: z.any().optional(), default_urgency: z.number().int().min(1).max(5).optional(), default_effort_estimate: z.number().min(0).optional(), default_next_step: z.string().optional() })`.

Locate the existing ticket creation form component. Based on Phase 2 work this is likely QuickAddInput.tsx or a CreateTicketModal. Read the actual file first to understand the current form structure before making changes.

**Template selector:** Add a "Use template" dropdown/combobox at the top of the ticket creation form (before the title field). Use `useQuery({ queryKey: ['templates'], queryFn: ... })` to populate options. The selector is optional — default state is "No template" / blank.

**Pre-fill on template select:** When the user selects a template from the dropdown:
1. Call `setValue('title', template.title, { shouldValidate: true })` — or leave title blank if template.title looks like a placeholder
2. Call `setValue('problem_statement', template.problem_statement, { shouldValidate: true })` if template.problem_statement is set
3. Call `setValue('urgency', template.default_urgency, { shouldValidate: true })` if set
4. Call `setValue('effort_estimate', template.default_effort_estimate, { shouldValidate: true })` if set
5. Call `setValue('next_step', template.default_next_step, { shouldValidate: true })` if set
6. Use `{ shouldValidate: true }` on ALL setValue calls to clear stale validation errors (RESEARCH.md Pitfall 5)

All pre-filled fields remain editable — user reviews and submits manually (per CONTEXT.md locked decision). The template selector should be clearable so user can reset to blank form.

**Same pattern for portal form:** If the portal intake form at /portal/[dept]/page.tsx also has ticket creation capability, add the same template selector there. The template_id does not need to be sent to the server — it is a client-side pre-fill helper only.

If QuickAddInput is too minimal for a template selector (e.g., it only has a title field), check if there is a more complete ticket creation modal/dialog from Phase 2. Add the template selector to the most complete ticket creation form component. Read the file before editing.
